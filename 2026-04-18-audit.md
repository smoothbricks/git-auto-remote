# git-auto-remote v0.7.0 hardening — agent-swarm prompt

**Status at time of writing**: HEAD = `7c68de8` (v0.6.3). Test suite: 184 pass / 0 fail across 15 files. Source: ~3900 LOC across `src/commands/` + `src/lib/`. Maintainer (Danny) is using this tool to drive a one-time ~600-commit private↔public mirror replay for `conloca/private` ↔ `conloca/conloca`. Cascading bugs in v0.5.x and v0.6.x have stalled that replay; v0.7.0 must be the final solid release that doesn't bring the maintainer back to file another patch.

## How we got here

This section is mandatory reading for any agent picking up this work. The bug pattern repeats and the next finding will fit it.

### v0.4.0 (baseline before this codebase was published-stable)

- `mirror pull` could replay commits from a disjoint-history mirror via `git am --3way`.
- Pause sub-cases existed but `am-continue` / `am-skip` were separate commands from review-pause `continue` / `skip`.
- v0.4.0 itself unified those into single `mirror continue` / `mirror skip` commands (the names current code uses).

### v0.5.0–v0.5.4 (sync configuration + bootstrap UX iterations)

- v0.5.0: introduced the `regeneratePaths` + `regenerateCommand` bucket. Renamed config from `fork-remote.*` → `auto-remote.*`. Breaking.
- v0.5.1: distinguished am structural failure from 3-way conflict so the UX stopped saying "resolve conflicts" when worktree was clean.
- v0.5.2: source-verbatim fallback when `applyReviewToWorktree` refused to apply via `--3way`.
- v0.5.3: root-inclusive bootstrap fix that turned out to infinite-loop on resume; **reverted in v0.5.4**.
- v0.5.4: bootstrap became OPTIONAL (`mirror pull` with no tracking ref does full-history replay).

### v0.5.5–v0.5.9 (mirror diff + classification + auto-apply iterations)

Each release shipped because the maintainer's replay hit a wall and the failure mode was MISSING from the test suite. Pattern: a code path that "worked in a happy-path test" produced wrong output in a real run because tests only exercised one classification shape.

- v0.5.5: `mirror diff` and `mirror source` subcommands. SHA format consistency.
- v0.5.6: 3 fixes: mirror diff scoped to source-commit buckets only, dropped redundant remote arg from pause footer, `mirror diff`/`source` bypass parseArgs so `--stat`/`--raw` forward correctly to git.
- v0.5.7: `mirror diff` shows review drift ONLY (not regenerate/outside). Reason: maintainer ran `mirror diff --stat` after a sub-case-B pause and got 5 unrelated files including a 2996-line `bun.lock` diff. `computeDiffPathspec` was returning `[...review, ...regenerate, ...outside]`. Test that should have caught it didn't exist; the test that DID exist for the previous release locked in the buggy behavior.
- v0.5.8: auto-apply partial commits when review bucket is empty (sub-case B). Reason: maintainer hit a pause that demanded `mirror continue` for zero staging work, on a commit whose review bucket was `[]` but had `outside` and `regenerate`. Pause was pointless.
- v0.5.9: same fix extended to sub-case C (included EMPTY, review EMPTY). v0.5.8 only covered sub-case B because the routing in `handlePartial` shorts to `handlePureReview` BEFORE the sub-case-B short-circuit fires. Same bug class as v0.5.8 but missed because tests didn't cover the sub-case-C entry path.

### v0.6.0–v0.6.3 (the sequence that drove this audit)

- **v0.6.0**: committer identity = author identity across all replayed commits. Added `commit-tree`-based rewrite after every `git am` in `applyRange`. Reason: maintainer noticed `git log --format='%cn'` showed his own name on every replayed commit instead of the original upstream author. The `git am` family preserves AUTHOR but not COMMITTER. v0.6.0 also added a runtime dep on `commander` later (v0.6.2 actually); pre-v0.6.2 used `node:util parseArgs`.

- **v0.6.1**: dropped `ensureMirrorRefspec` auto-add of mirror fetch refspec. Reason: `mirror pull` silently fast-forwarded the local tracking ref via fetch clobber. The tool itself had been auto-adding `+refs/git-auto-remote/mirror/*:refs/git-auto-remote/mirror/*` to every mirror remote's fetch refspec on every pull, intending to "replicate tracking ref state server-side" — but the fetch direction is the OPPOSITE of replication. Result: every fetch force-overwrote local tracking with whatever the remote happened to have, silently skipping in-range commits. v0.6.1 stopped the auto-add but did not clean up existing config or instruct CI scripts to stop adding it.

- **v0.6.2**: commander-powered help + per-subcommand `--help` + cross-direction-leak warnings + strict flag parsing. Reason: maintainer typed `mirror status --remote` (singular) and the tool silently swallowed it (`parseArgs strict: false`). Investigation revealed:
  1. Help text was a hand-typed monolithic string with eyeballed column padding.
  2. `--remotes` flag was undocumented in USAGE.
  3. **CRITICAL**: maintainer's local config had `remote.public-repo.push = refs/git-auto-remote/mirror/*:refs/git-auto-remote/mirror/*` from earlier session work. This pushed the cross-direction tracking ref (`mirror/private-repo/last-synced` pointing at a private commit) to `conloca/conloca` (the public repo). git-push transferred the FULL OBJECT CLOSURE — 565 private commits including all `privpkgs/*` source — to the public repo's object DB, accessible to anyone via `git fetch <url> 'refs/git-auto-remote/*:refs/git-auto-remote/*'`. The leak was fixed by deleting the ref + narrowing the push refspec to same-direction-only. v0.6.2 added prominent SECURITY warnings in `mirror-bootstrap` docstring + `mirror-status` docstring + README.

- **v0.6.3**: `mirror skip` enforces tracking-ref invariants. Reason: maintainer ran `mirror skip` to drop a commit they didn't want; "Skipped: 5a1ec3a4 ..." printed, then immediately the SAME commit re-applied. Loop forever. Root cause: skip's review-pause / pure-review-pause / am-in-progress-fallback paths called `clearReviewPending + reset HEAD~1 + return mirrorPull(remote)` but did NOT call `updateTrackingRef`. They TRUSTED that the post-applypatch hook had advanced tracking during the original `git am`. When ANY external thing perturbed the ref between pull-pause and skip (in this case a leftover `+refs/git-auto-remote/mirror/*:...` fetch refspec in `.git/config` that the resume's fetch honored), tracking got rewound and the same commit re-appeared. v0.6.3 fix: every skip path now explicitly `updateTrackingRef(remote, review.sourceSha)` before tail-calling `mirrorPull`. PLUS `mirrorPull` switched to passing an EXPLICIT narrow refspec (`+refs/heads/<syncBranch>:refs/remotes/<remote>/<syncBranch>`) to `git fetch`, making the tool's own fetch immune to misconfigured `remote.<X>.fetch` entries.

### The pattern

Every release in v0.5.x–v0.6.3 fits one of two categories:

1. **"Trust prior state" defects**: the code assumes some external invariant holds (post-applypatch fired, fetch didn't clobber, classification produces consistent output) and calls a downstream function without re-asserting it. Tests cover happy paths where the invariant DOES hold; perturbation paths are missing.

2. **"Untested classification shape" defects**: a classify routing decision (sub-case A vs B vs C, included-empty vs review-empty vs regenerate-only, etc.) introduces a new code path that isn't covered. Bugs surface when a real-world commit hits the missing path.

The `2026-04-18-audit.md` (this file) and the companion `AUDIT-v0.6.3.md` enumerate every additional instance of these two patterns currently present in the codebase. **v0.7.0 must close all of them.**

### Batch 2 — `postAmTransition` HEAD verification (CRIT-2)

**Reference**: `AUDIT-v0.6.3.md` lines 75-88.

**Files affected**:

- `src/commands/mirror-continue.ts` — `postAmTransition` function (line ~150)
- `test/mirror-resume.integration.test.ts` — new test for am-bypass scenario

**Defect**: `postAmTransition` overlays review content on HEAD and sets `phase: 'review-pause'` without verifying HEAD's tree actually matches `sourceSha`'s included subset. If the user bypassed the wrapper (`git am --abort` directly, or manual `reset --hard HEAD~1` between conflict resolution and `mirror continue`), HEAD is no longer at the expected state. Tool then overlays review against wrong HEAD silently.

**Fix approach**: At the entry to `postAmTransition`, verify HEAD's tree matches the included-subset tree of `sourceSha`. The included subset is the source's tree filtered by syncPaths. Compute the expected tree (or use a cheaper proxy: list the included paths' blobs in HEAD and compare to source). If mismatched, refuse with an explanatory error: "HEAD does not contain the just-applied subset of <sha8>. The git am state may have been bypassed (git am --abort, manual reset, etc.). Resolve manually or run 'mirror skip <remote>' to drop the source commit."

**Tests to add**:

1. Sub-case A pause (am conflict). User runs `git am --abort` directly. Marker still says `phase: 'am-in-progress'` but `amInProgress() === false` and HEAD is back at scaffold. Run `mirror continue`. Assert: tool refuses with the explanatory error AND does NOT silently set phase: 'review-pause'.

2. Same setup but user runs `git reset --hard HEAD~1` after the am succeeded (somehow). Same assertion.

Verify both tests fail on `7c68de8` (current code silently transitions to review-pause).

### Batch 4 — `post-applypatch` validation hardening (CRIT-3 + MEDIUM-4)

**Reference**: `AUDIT-v0.6.3.md` lines 90-104, 198-202.

**Files affected**:

- `src/commands/post-applypatch.ts` — every `return 0` validation branch must do BOTH: log a warning to stderr explaining what was malformed AND call `clearMirrorInProgress()` to prevent stuck-sentinel
- `src/commands/mirror-pull.ts` — handle the stuck-sentinel state (sentinel set + `amInProgress() === false`)
- `test/post-applypatch.test.ts` — extend with all defensive branches

**Defect 1**: `post-applypatch` silently no-ops on every parse failure. If the patch's `From <sha>` header is malformed, or the `next`/`last` files are corrupted, the hook returns 0 without advancing tracking AND without clearing the sentinel. Result: tracking stops advancing mid-range with no signal; sentinel can get stuck.

**Defect 2**: No code path handles `amInProgress() === false ∧ getMirrorInProgress() !== null` (stuck sentinel). `mirrorPull`'s precondition check at line 91-93 looks for `amInProgress()` only.

**Fix approach**:

1. In `post-applypatch.ts`, replace silent `return 0` with `console.error('[post-applypatch] <reason>; skipping')` for each unexpected case. Always attempt to clear sentinel if we've decided we can't make progress on this patch.
2. In `mirror-pull.ts` precondition block, add a check: if sentinel is set BUT `amInProgress()` is false, REFUSE with this exact message and exit 1: `[git-auto-remote] stale mirror-in-progress sentinel for <remote> but no git am running. Run 'git-auto-remote mirror skip <remote>' to clean up.` Do not auto-clean — the user might have rebased mid-pause and silent cleanup would mask the inconsistency.

**Defect 3 (MEDIUM-4)**: `post-applypatch.ts:51` uses `next >= last` to decide whether to clear the sentinel. The `>` case (impossible in normal git operation but possible under bizarre abort/retry states) currently clears anyway. Add a comment justifying the defensive `>=` choice OR change to `===` and surface the unexpected-state case explicitly.

**Tests to add**:

1. Set sentinel + write malformed `rebase-apply/next` (e.g. `"abc"`). Call `postApplypatch()`. Assert: tracking unchanged AND sentinel cleared (or a stderr warning emitted).
2. Set sentinel + missing `rebase-apply/last`. Call `postApplypatch()`. Assert: same.
3. Set sentinel + patch file's first line lacks `From <40hex>`. Call `postApplypatch()`. Assert: same.
4. New test: sentinel set + `amInProgress() === false`. Call `mirrorPull`. Assert: exit 1 + stderr contains exact message `[git-auto-remote] stale mirror-in-progress sentinel for <remote> but no git am running. Run 'git-auto-remote mirror skip <remote>' to clean up.`

# Part II — Swarm execution model

This section describes how to deploy hundreds of parallel agents against the v0.7.0 scope without conflicts.

## Coordination overview

The work is organized into **5 phases**, each containing 1..N **batches**, each containing 1..N **tasks**. Tasks within a batch are SERIAL (operate on the same file). Batches within a phase are PARALLEL. Phases are SEQUENTIAL (Phase N+1 starts only after Phase N's barrier closes).

```
Phase 0: Bootstrap                    [1 agent, blocking]
   |
   v
Phase 1: Parallel scaffolding         [N agents, fully parallel - new files only]
   |
   v
Phase 2: Per-file fix batches         [M agents, parallel across batches; serial within]
   |
   v
Phase 3: Integration verification     [1-3 agents, parallel post-merge checks]
   |
   v
Phase 4: Release commit + tag         [1 agent, blocking]
```

## Worktree-per-agent isolation (LOCAL-ONLY, NO PUSH)

**Every agent works locally. No `git push` ever. The maintainer handles all remote operations manually after the swarm finishes.**

To eliminate filesystem contention between parallel agents, each batch runs in its OWN git worktree on its OWN local branch. P0 creates all per-batch branches upfront so agents never branch concurrently.

```sh
# Phase 0 setup (P0-BOOTSTRAP runs this verbatim)
cd /Users/danny/Dev/git-auto-remote
git checkout main
git checkout -b v0.7.0-integration   # local only

# P0 creates one branch + worktree PER BATCH (not per task):
for B in T1-NH T1-NB T1-SKEW T2-MCONT T2-MPULL T2-PAP T2-MSTAT T2-MSKIP T2-MDIFF T2-MSRC T2-MCFG T2-REGEN T2-HOOKS T2-MMSC; do
  git branch v0.7.0-$B v0.7.0-integration
  git worktree add /tmp/gar-$B v0.7.0-$B
done
```

Each batch agent works inside its assigned `/tmp/gar-<batch>` worktree on its assigned `v0.7.0-<batch>` branch. Worktrees share the object DB; each has its own working tree and HEAD; no two agents ever check out the same branch.

Test fixtures already use `mkdtempSync(join(tmpdir(), 'gar-...'))` so test-side fs is also isolated.

After all batches complete (DONE in TASK_LOG.md), Phase 3 cherry-picks each batch's commits into `v0.7.0-integration`. **Always cherry-pick, never merge.** This produces a linear history.

After Phase 4 tags `v0.7.0` (local only), maintainer (Danny) does final review, decides whether to publish to npm + push to origin. The swarm does not.

## TASK_LOG.md coordination

A single append-only log at `/Users/danny/Dev/git-auto-remote/TASK_LOG.md` tracks status. Each agent on start appends:

```
T<ID> START <ISO-timestamp> <agent-id>
```

On completion or failure:

```
T<ID> DONE  <ISO-timestamp> <agent-id> <branch-or-commit-sha>
T<ID> FAIL  <ISO-timestamp> <agent-id> <reason>
T<ID> BLOCKED <ISO-timestamp> <agent-id> <waiting-on-T-ID-or-reason>
```

Phase 3 integration agent reads the log to know which task branches to merge. If any T<ID> shows FAIL or BLOCKED without a subsequent DONE, escalate to maintainer rather than tagging v0.7.0.

The log file MUST be appended atomically (`echo "..." >> TASK_LOG.md` is fine on POSIX; never overwrite). Two agents writing simultaneously risk interleaved bytes but not lost lines under typical filesystem semantics.

## Coordination rules each agent must follow

1. **Worktree exclusivity**: never `cd` outside your assigned worktree. The shared object DB makes file-level operations safe; cross-worktree HEAD operations are not.

2. **No global git config writes**: any `git config --global` is forbidden. Only `git config --local` (which writes to YOUR worktree's `.git/config` — actually no, `git config --local` writes to the SHARED `.git/config` because worktrees share it. Use `git -c <key>=<val>` per-invocation OR `git config --worktree` if needed). For test fixtures, use temp git repos created by your test setup, not the actual `git-auto-remote` repo.

3. **No npm/bun add**: do not add or remove dependencies. v0.7.0 stays on commander @14.0.3 + dev deps as-is.

4. **No package.json edits**: Phase 4 owns the version bump.

5. **No history rewriting**: no `git rebase --interactive`, no `git commit --amend` after committing. If you need to reorder, abort and let P3-INTEGRATE handle it.

6. **Source-of-truth read order**: this file first → `AUDIT-v0.6.3.md` second → relevant existing tests third → relevant source last. Do not modify source until you have read its callers.

7. **Documentation updates**: every fix that changes a contract MUST update the relevant docstring AND a code comment referencing the audit finding ID:

   ```ts
   // v0.7.0 CRIT-N (see 2026-04-18-audit.md): <one-line rationale>
   ```

8. **Test diagnostic clarity**: assertion messages should make the failure mode obvious. Prefer `expect(value).toBe(expected)` with a clear value over `expect(condition).toBe(true)`. For console-capture tests, prefer `expect(captured).toContain(specific-string)` plus `expect(captured).not.toContain(other-specific-string)` over single-direction asserts.

9. **No silent regressions**: if `bun run check` reports any test that wasn't failing on the baseline, you broke it — fix it before committing.

10. **Final-state cleanup**: when your task is DONE, your worktree should be clean (`git status` empty, except for files you intentionally created/modified that are tracked in the commit).
